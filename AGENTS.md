# 开发流程 Rule

本文件约束本项目后续所有开发、重构、修复和文档更新工作。每次开始改动前先阅读本规则，并在提交前逐项确认。

## 1. 内容归位

- 路由相关代码放在 `routes/`，只负责请求解析、参数校验、响应封装和薄层委托。
- 可独立测试的业务逻辑放在 `services/`，保持纯 Python 逻辑，不直接依赖 Flask 请求对象。
- 数据访问、SQL 查询、事务和持久化逻辑放在 `repositories/`，不得混入 HTTP 或页面逻辑。
- 对外统一仓储入口放在 `models.py` 的 `InventoryRepository`，新增仓储能力需要通过该外观层暴露。
- 配置项统一放在 `config.py`，支持环境变量覆盖时应沿用现有命名和默认值风格。
- 数据库表、索引和兼容迁移统一放在 `init_db.py`，不得散落在路由或服务代码中。
- 前端页面、样式和脚本放在 `static/`，CLI 客户端能力放在 `inventory_client/` 和 `inventory_cli.py`。
- 测试放在 `tests/`，测试文件命名遵循 `test_*.py`，并复用 `conftest.py` 中的 fixtures。
- 项目说明、架构说明和开发约定放在 `README.md`、`CONTRIBUTING.md`、`docs/ARCHITECTURE.md` 或 `docs/` 下的专题文档。

## 2. 文件中文注释

- 每个新增或重写的源代码文件，都必须在文件顶部写明中文文件说明，说明该文件的职责、所属层级和主要边界。
- 每个新增函数、类或复杂代码块，应补充简洁中文注释或 docstring，说明用途、输入输出和关键约束。
- 注释必须解释业务意图、结构边界或非显而易见的实现原因，避免只重复代码表面含义。
- 修改既有文件时，如果文件缺少清晰的中文职责说明，应在本次改动范围内补齐；不做无关的大范围格式化。

## 3. 结构完整性

- 新功能必须沿现有分层流向接入：`routes/` -> `services/` -> `models.py` -> `repositories/` -> SQLite。
- 不得为了局部便利绕过既有外观层、仓储层或配置层。
- 新增模块时，要同步考虑导出入口、调用路径、错误处理、测试覆盖和文档位置。
- 删除或移动代码前，要确认调用方、测试、文档和静态资源引用已经同步更新。

## 4. 可读性

- 命名应准确表达业务含义，避免使用含糊缩写。
- 单个函数保持职责单一；当参数、分支或副作用明显增多时，应拆分为小函数或服务函数。
- 错误信息、日志信息和用户可见文案优先使用清晰中文，API 字段名保持现有英文风格。
- 新增逻辑应优先复用已有工具函数、响应封装、仓储基类和测试 fixtures。

## 5. 可维护性

- 变更数据库结构时，必须在 `init_db.py` 中提供兼容迁移逻辑，并补充对应测试。
- 变更 API 行为时，必须更新或新增测试，覆盖成功路径和关键失败路径。
- 变更前端交互时，必须保持现有页面结构和资源组织清晰，不把大量业务逻辑堆进 HTML。
- 不引入与当前技术栈不一致的新框架或重型依赖，除非能明确降低复杂度并同步更新文档。

## 6. 文档同步

- 每次代码更改完成后，必须检查并更新 `docs/ARCHITECTURE.md`。
- 如果变更涉及目录结构、模块职责、数据表、API、认证、业务流程、测试方式或部署方式，必须在 `docs/ARCHITECTURE.md` 中同步记录。
- 如果本次变更不影响架构文档内容，也要在提交说明或开发记录中明确说明“已检查 ARCHITECTURE，无需更新”。
- 新增开发规则或流程约定时，优先更新本文件，并在 `CONTRIBUTING.md` 或相关 `docs/` 文档中添加入口。
